🏗 Architecture: EVM to UTXO

Original (EVM - Abstract Chain)

5 Solidity Contracts Registry.sol - Membership via heartbeat CouncilSeats.sol - Seat auctions, voting power Governance.sol - Proposal lifecycle Forum.sol - Threads, comments, petitions Treasury.sol - Fund custody + execution Storage Model: Contract state (mappings) Execution: Sequential state mutations Fees: Gas-based (variable) MEV: Possible (sequencer ordering)

5 Aiken Validators (10,746 bytes total) registry.ak (1,553b) - Member UTxOs + config council.ak (2,079b) - Auction/seat/refund UTxOs governance.ak(3,554b) - Proposal/vote UTxOs forum.ak (2,637b) - Thread/comment/petition UTxOs treasury.ak ( 923b) - Fund custody UTxOs Storage Model: Individual UTxOs with datums Execution: Deterministic, parallel Fees: Flat, predictable MEV: Impossible (UTXO model)

🐛 Bugs Found During EVM-to-UTXO Translation
Bool Encoding (CBORTag)
False = constructor 0 (CBORTag 121), True = constructor 1 (CBORTag 122). Counterintuitive and caused silent validator failures.
Slot vs POSIX Time
Aiken validators receive POSIX milliseconds from validity_range, not slot numbers. Off-chain must convert: posix_ms = (genesis_epoch + slot) * 1000.
MinUTxO with Datums
Datum size increases minimum UTxO requirement. Membership datum (PKH + 3 integers) needs 3 AP3X, not the default 2 AP3X.
PlutusV3 Option<T> Auto-Wrapping
PlutusV3 runtime auto-wraps on-chain datum in Some(). Store raw type (Int, ByteArray), NOT manually wrapped in Option constructor.
Aiken Blueprint Bug (Custom Records)
Adding custom record types to multi-validator Aiken projects causes silent compilation failure. Workaround: use primitive types.
🧪 Advanced Capabilities Proven on Vector
Full Vote Lifecycle (COMPLETE)
Submit -> Deliberation -> Voting -> CastVote(For) -> CastVote(Against) -> FinalizeVote(Defeated). 6 governance redeemers tested. Quorum + threshold math proven on-chain.
CIP-31 Reference Inputs
Read data from 3 different script addresses without consuming UTxOs. Enables composable cross-validator reads (EVM: cross-contract calls).
CIP-32 Inline Datums
All 19 UTxOs across 5 validators use inline datums. Combined with CIP-31, this enables full cross-validator state reading.
Auction + Settlement Cycle
Place bid -> Outbid with refund -> Settle auction -> Create council seat. Full auction lifecycle with multi-output UTxO patterns.
Membership Pruning
Remove expired memberships and reclaim UTxOs. POSIX time comparison for expiry detection.
Multi-Agent Interaction
5 independent agent wallets interacting with shared validators. Concurrent P2P transfers + script spending.
