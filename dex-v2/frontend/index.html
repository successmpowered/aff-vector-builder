Architecture
DEX V2 uses three PlutusV3 validators working together

limit_order

Holds individual orders as UTxOs. Each UTxO contains the offered tokens and an inline datum specifying the minimum acceptable receive amount, the owner, and a unique order ID. Anyone can consume (fill) an order by satisfying the min_receive constraint.

amm_pool

Manages liquidity pools for automated market making. Pools hold reserves of two token types and enforce the constant-product invariant (x * y = k). Swaps pull from pool reserves and pay the LP fee on-chain.

matcher

Coordinates multi-order matching in a single transaction. The matcher validator verifies that every filled order in a batch received at least its min_receive, enabling atomic settlement of multiple orders with no partial fills.

How It Works

The DEX V2 order book is fully on-chain and permissionless. When a user places a limit order, they send tokens to the limit_order script address with an inline datum describing the trade parameters. The order sits as a UTxO until someone fills it.

Any wallet can fill an order — there is no registration, no staking requirement, and no centralized batcher. The filler constructs a transaction that consumes the order UTxO and sends the owner at least min_receive tokens. The on-chain validator enforces this constraint, making it trustless.

MEV resistance comes from three properties: flat scooper fees (no bidding wars), UTXO determinism (transactions are fully determined before submission), and on-chain price enforcement (validators check min_receive, preventing sandwich attacks).
